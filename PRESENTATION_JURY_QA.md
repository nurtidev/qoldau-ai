# Qoldau AI — Q&A для жюри хакатона

> Заготовки ответов на типичные вопросы. Структура: вопрос → короткий ответ (для устной речи) → опорные факты для углубления (если жюри потянет нить).

## Блок 1. Технические вопросы

### 1.1. Где гарантия, что это no-code, а не хардкод под видом конструктора?

Откройте `psql` и выполните `SELECT title, jsonb_array_length(form_schema->'steps') FROM services;` — увидите, что вся структура шагов, полей, формул, условий лежит в JSONB-колонке. Код [FormRenderer/index.tsx](frontend/src/components/FormRenderer/index.tsx) один на все 16 услуг и не содержит ни одного `if title == 'Лизинг'`.

### 1.2. Что произойдёт, если я попрошу администратора создать совершенно новую меру прямо сейчас?

Зайдёт в **Администрирование → Услуги → Создать**. Введёт название и описание. Нажмёт «Сгенерировать форму через AI» — Anthropic Claude вернёт JSON-схему за ~5 секунд. Дальше визуально подправит поля и нажмёт «Опубликовать». Услуга появится в каталоге без редеплоя.

### 1.3. Что если AI сгенерирует кривую форму?

AI-результат — черновик, не финальный. Администратор открывает визуальный конструктор и правит руками: добавить/удалить поля, поменять типы, прописать формулы, условия. AI снижает время с часов до минут, но человек принимает решение.

### 1.4. Какая модель AI? Почему именно она?

Anthropic Claude, идентификатор `claude-sonnet-4-6`. Выбор по соотношению качества структурированного JSON-вывода и стоимости. Возможна замена на любую другую модель в [backend/internal/handlers/ai.go](backend/internal/handlers/ai.go) без изменений фронтенда.

### 1.5. Расчётные поля — как именно работают?

Поле `calculated` имеет `formula` — JS-выражение, где переменные = `id` других полей. Пример из контрольного кейса лизинга: `f_total_cost * f_advance_pct / 100`. Пересчёт идёт на клиенте в реальном времени при изменении любого зависимого поля. Сервер при сохранении заявки проверяет результат.

### 1.6. Безопасность формул — что если в формулу запихнут вредоносный код?

Формулы создаёт только администратор, не пользователь-заявитель. Вычисление происходит на клиенте в ограниченном контексте. Для продакшна следующий шаг — sandbox-вычислитель (например, mathjs или whitelisted-AST), чтобы исключить произвольный JS даже от администратора.

### 1.7. Стек и почему такой?

Frontend — React 18 + TypeScript + Vite, TanStack Query v5, Zustand. Backend — Go 1.22 + chi + sqlx. БД — PostgreSQL 16 (JSONB критичен для form_schema). Redis для сессий. Деплой — docker-compose, в проде nginx как SPA-server + API-proxy. Выбор по скорости итерации (Vite/TS) и устойчивости backend (Go).

### 1.8. Производительность — что будет на 100 000 заявок?

JSONB в Postgres индексируется GIN-индексами, поиск по полям внутри JSON работает на десятках миллионов строк. Узкое место не БД, а интеграции с внешними системами (eGov, КГД, ПКБ) — там нужны очереди и retry. Сейчас это не построено, потому что mock.

### 1.9. Условная видимость шагов — насколько глубоко?

Operator: equals / not_equals / greater_than / less_than. Условие висит на шаге или на поле. Пример: шаг «Аудиторское заключение» появляется, только если `f_org_type = Крупный бизнес`. Вложенных условий пока нет — для контрольного кейса не понадобилось.

### 1.10. Eligibility-движок — это что отдельно?

Отдельная JSONB-колонка `eligibility_rules` в таблице `services` (миграция [004_eligibility_rules.up.sql](backend/migrations/004_eligibility_rules.up.sql)). Правила: `blocking` (стоп) и `warning`. Проверяются до подачи заявки по данным формы + mock-КГД. Цель — заявитель видит риск отказа заранее, не ждёт месяц решения.

### 1.10a. Калькулятор охвата — это какая фича?

Сверх ТЗ. Симметричная eligibility-движку обратная задача: вместо «подходит ли *этот* предприниматель под программу» — «*сколько* предпринимателей подходят». В конструкторе формы — режим «Аудитория» (миграция [005_audience.up.sql](backend/migrations/005_audience.up.sql), [handlers/audience.go](backend/internal/handlers/audience.go)). Админ настраивает фильтры (отрасль, регион, размер бизнеса, выручка, возраст бизнеса, исключения по налогам/реестру), видит число подходящих за ~300 мс плюс разрезы по регионам / отраслям / МСБ. Кнопка «Отправить уведомление» → персонализированная рассылка с подстановкой `{{full_name}}` и `{{org_name}}` каждому. Источник в MVP — 3000 синтетических профилей с распределением по РК; в продакшне — eGov + КГД + ИС МСБ через ЕИШ.

### 1.10b. Откуда такие распределения профилей в синтетической базе?

Приближенно по реальной структуре МСБ в РК: г. Алматы 20%, Астана 15%, Шымкент 8%, регионы 57%. Отрасли: торговля 30%, агро 20%, услуги 15%, производство 12%, остальные. Выручка: микро (<30 млн ₸) 50%, малый (30–300 млн) 30%, средний (300М–3 млрд) 15%, крупный 5%. Возраст бизнеса: 0–12 мес 20%, 12–36 мес 25%, 36–120 мес 30%, >120 мес 25%. Налоговая задолженность 8%, реестр риска 2%. Сидинг детерминирован через `setseed(0.4242)` — каждый запуск миграции даёт идентичные 3000 записей.

### 1.10c. Что значит «персонализированная»? Это не email-маркетинг?

Уведомление складывается в `notifications` таблицу того пользователя, который попал под фильтр. Он видит его в личном кабинете (там же, где статусы заявок). Подстановка `{{full_name}}` и `{{org_name}}` происходит на сервере при INSERT — каждый получатель видит свой текст. Email/SMS-канал не реализован — это интеграционная задача после хакатона. Для B2G-портала уведомление в личном кабинете — основной канал, не email.

### 1.11. Тесты?

Unit-тесты на бизнес-логику отдельных хендлеров — частично. End-to-end сценария «администратор → пользователь → решение» нет, проверяется руками по сценарию из README. Для продакшна — приоритет.

## Блок 2. Бизнес-модель и коммерция

### 2.1. Как зарабатываете?

На текущем горизонте — два сценария. Первый: внедрение в институт развития как поставщик SaaS / on-premise решения, оплата по годовой лицензии. Второй: контракт по линии цифровой трансформации (бюджет ведомства / акимата). Прямой монетизации с предпринимателей не планируется — для них портал бесплатен.

### 2.2. Сколько стоит внедрение?

Зависит от объёма пилота. Диапазон для пилота одного института развития с одной–тремя мерами — 10–50 млн ₸ за 60–90 дней, конкретная сумма — после технического согласования. Цифру одной строкой называть отказываюсь, в переговорах это становится потолком.

### 2.3. Чем отличается от e.gov.kz?

e.gov.kz — это шлюз доступа к услугам с фиксированной структурой каждой услуги, изменения проходят через цикл разработки. Qoldau AI — это конструктор форм для самих провайдеров услуг (БРК-Лизинг, Damu и т.д.), который позволяет ведомству менять структуру и параметры заявки без программистов. Это не замена e.gov, это слой администрирования над ним.

### 2.4. Чем отличается от Typeform / Tally / Google Forms?

Спец. функции под госуслуги: расчётные поля с финансовыми формулами (DSCR, IRR, аннуитет), условная видимость по статусу субъекта (МСБ/крупный), префилл из eGov-источников по ИИН, eligibility-движок с проверкой по КГД, калькулятор охвата программы по реальным профилям предпринимателей в системе, персонализированные адресные уведомления, интеграция с учётными системами институтов развития. Typeform этого не умеет и не должен.

### 2.4a. Какую боль решает калькулятор охвата на стороне Damu/БРК-Лизинг?

Сегодня институт развития запускает программу и узнаёт фактическое число потенциальных получателей **по поступившим заявкам ретроспективно** — за квартал или дольше. Если фильтры программы заданы слишком узко, об этом понимают, когда поток заявок маленький. Калькулятор охвата отвечает на вопрос **до запуска**: «при таких параметрах под программу подпадает X компаний, по регионам так-то». Плюс адресная рассылка: вместо пассивного каталога — программа сама находит свою аудиторию.

### 2.5. Кто конкурент в РК?

Прямого конкурента-конструктора для гос. мер поддержки в публичном поле не вижу. Альтернативы: разработка под каждую меру отдельно (текущая практика большинства институтов) либо адаптация западных low-code платформ (OutSystems, Mendix) — это в разы дороже и с обязательной зарубежной зависимостью.

### 2.6. Размер рынка?

70+ мер господдержки в техническом задании хакатона — это срез одного периметра (Байтерек + дочерние). По всей РК — институтов развития, ведомств и акиматов, у которых есть услуги с заявкой и принятием решения, на порядок больше. Точную TAM-цифру не называю — это будет нарисованная цифра без источника.

### 2.7. Юнит-экономика?

Пока pre-traction, юнит-экономики нет. После пилота — мерять стоимость подключения одной меры (часов администратора), время от описания до публикации, стоимость инфраструктуры на одну заявку. Это ключевые метрики для следующего этапа.

## Блок 3. Команда

### 3.1. Кто стоит за проектом?

Один разработчик. Full-stack (Go + React/TS), интеграция AI (Anthropic), миграции БД, инфраструктура (Docker), архитектура — реализовано одним человеком за период подготовки к хакатону.

### 3.2. Почему не команда?

Подготовка к хакатону вёлась индивидуально. На стадии MVP такой формат позволяет двигаться быстрее: нет накладных расходов на коммуникацию, синхронизацию, кодовые конвенции. На стадии пилота с институтом развития команда расширится — нужен будет интеграционный инженер и ИБ-специалист как минимум.

### 3.3. Кого нужно нанять для пилота?

Backend-инженер с опытом интеграций (eGov, КГД, ЭЦП NCALayer) и специалист по ИБ для прохождения аттестации соответствия. Возможно — продуктовый аналитик из периметра института развития для перевода регламентов в формы. Конкретный план найма — после получения пилотного контракта.

## Блок 4. Пилот и интеграции

### 4.1. Сколько времени до реального пилота?

60–90 дней с момента подписания договорённости. Этапы: техническое согласование (2 недели), интеграция с одной учётной системой выбранной меры (4–6 недель), ИБ-проверка и согласие на ПДн (2–3 недели), мягкий запуск на ограниченном контуре получателей.

### 4.2. Что нужно от института развития для пилота?

Четыре пункта: (1) куратор со стороны института, (2) одна мера как пилотная (предпочтительно — массовая, для статистики), (3) технический доступ к учётной системе приёма заявок, (4) согласие на использование тестового eGov-контура. Финансирование и разработку беру на себя.

### 4.3. Что делать с реальным eGov-входом?

Текущий mock-эндпоинт `/api/mock/egov/:iin` заменяется на интеграцию через ЭЦП NCALayer и шлюз eGov. Это типовая интеграция, проходится за 3–4 недели, основное время — оформление документов, не код.

### 4.4. Сертификация по ИБ — есть?

Сейчас — нет. Это часть пилотного этапа: аттестат соответствия ИБ-требованиям РК, аудит обработки ПДн (Закон РК № 94-V). Без сертификата система не имеет права обрабатывать реальные ИИН в продакшне. В рамках хакатона все ИИН — тестовые / mock.

### 4.5. А если институт развития уже автоматизировал свою услугу?

Тогда мы дополняем, а не заменяем. Qoldau AI — слой администрирования и подачи. Учётная система института остаётся, мы передаём в неё подготовленную и провалидированную заявку через интеграцию. Меньше работы для института, лучше качество входящих заявок.

## Блок 5. Неудобные вопросы — отвечать прямо

### 5.1. Почему solo? Это же признак слабого проекта.

Solo на стадии MVP — это скорость и владение каждым решением. Это становится слабостью на стадии масштабирования — поэтому первый шаг после хакатона — расширение команды до 3–4 человек с интеграционными и ИБ-компетенциями. Я не считаю это секретом или проблемой — это фаза.

### 5.2. У вас нет реальных интеграций. Что вы вообще показываете?

Показываю работающий движок форм, реалистичный контрольный кейс уровня БРК-Лизинг (DSCR / IRR / KYC / казсодержание), AI-генерацию формы из текстового описания и архитектуру, которая масштабируется на 70+ мер. Интеграции — следующий этап, и они невозможны без доступа к данным институтов развития, который как раз даёт пилот.

### 5.3. Откуда уверенность, что Байтерек / Damu возьмут вас в пилот?

Уверенности нет — есть гипотеза. Гипотеза проверяется именно встречей с куратором после хакатона. Если ответ «нет, не возьмём», это снимает претензию проекта на этот сегмент, и я перенесу фокус на акиматы или отраслевые ведомства. Это управляемый риск.

### 5.4. Что если у вас украдут идею?

Идея no-code-конструктора форм — не уникальна и не патентуема. Защита — в скорости исполнения, доменной экспертизе по конкретным мерам поддержки в РК и в первом пилоте, который даёт преимущество входа. Если идею «украдёт» условный СЦ Байтерек — это будет, по сути, внедрение того же подхода, что и моё, и тогда задача техзадания всё равно решена.

### 5.5. 16 услуг — это не 70+. Где остальные?

16 — это засеянные демонстрационные услуги с реалистичной финансовой моделью каждая. 70+ — это целевая ёмкость архитектуры, перечислены в техническом задании хакатона. Засевать ещё 50 услуг руками без согласования с институтами развития — это будет пустая декорация. Реальные 70+ заполняются после пилота — администраторами институтов через тот же конструктор.

### 5.6. Что если Anthropic заблокирует API в РК?

AI-генерация — вспомогательная функция, не критическая. Без неё конструктор продолжает работать руками. Альтернативные провайдеры (OpenAI, Yandex, локальные модели) подключаются заменой одного хендлера. На уровне архитектуры завязки на Anthropic нет.

### 5.7. Почему это лучше, чем «купить готовое western low-code»?

Три причины. Первая: цена — лицензия OutSystems/Mendix на нужный объём обойдётся в десятки раз дороже годовой инфраструктуры Qoldau. Вторая: специфика — формулы для DSCR, казсодержания, согласия по Закону № 94-V, ЭЦП NCALayer — это не «настраивается», это «пишется». Третья: суверенитет данных — данные граждан РК на инфраструктуре в РК, без зарубежной зависимости.
