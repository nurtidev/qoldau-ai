> Раунд ЕППБ · подача 12.07.2026 · финалисты 14.07.2026 · Demo Day 16.07.2026 (alem.ai) · Russian language · solo-команда

## Слайд 1 — Что это

**Qoldau AI — единый портал поддержки бизнеса (ЕППБ), где каждая мера господдержки задаётся как форма в универсальном no-code конструкторе, а не пишется в коде.**

Контекст: техническое задание раунда требует реализовать услуги через универсальный конструктор форм. Иначе задание не зачитывается. Это ядро проекта: один движок форм + один конструктор + одна база данных обслуживают все меры — от лизинга вагонов в БРК-Лизинг до финансирования животноводства в КазАгроФинанс. Заявитель проходит услугу в одном окне: подача с eGov-префиллом → предоценка A–D → отправка → **дозапрос документов** → досдача этапа 2 → решение.

Аналог по подходу: Typeform / Tally — но для государственных мер поддержки, с расчётными полями, условной видимостью, двухэтапной подачей, mock eGov-префиллом, предоценкой заявителя и AI-генерацией формы из текстового описания.

## Слайд 3 — Что реально работает

Сначала — то, что прямо требует ТЗ раунда (конструктор, двухэтапность, обе контрольные услуги, AI, карта/аналитика/материалы). Затем — модули клиентского пути и продуктовой аналитики.

**Модуль 1. Универсальный конструктор форм (для администратора дочки холдинга)**
- Создание услуги: название, описание, категория, орган-провайдер, headline-параметры (ставка / сумма / срок)
- Визуальный редактор шагов и полей: text / textarea / number / currency / select / multiselect / date / file / checkbox / radio / calculated
- Расчётные поля с формулами — переменные = `id` других полей, вычисление через `expr-eval` (без `eval`)
- Условная видимость шагов и полей (operator: equals / not_equals / greater_than / less_than)
- **Переключатель этапа (stage 1 / stage 2) на каждом шаге** — разметка, какие шаги идут в первичную подачу, а какие в дозапрос документов
- Префилл из mock eGov по ИИН / БИН / названию; публикация без редеплоя; онбординг-тур (`react-joyride`)

**Модуль 2. Двухэтапная заявка (главное новое требование ТЗ)**
- Статусная модель: `submitted → in_review → [docs_requested] → approved | rejected`
- Первичная подача — шаги `stage 1`; `POST /api/applications` со статусом `submitted`
- Дозапрос: администратор ставит `docs_requested` + `request_message` (что донести) → уведомление заявителю
- Досдача: заявитель заполняет шаги `stage 2` (обычно документы), `POST /api/applications/:id/stage2` — `SubmitStage2` **мёржит** данные поверх этапа 1, переводит `stage → 2`, `status → in_review`
- Одна запись `applications`, одна JSONB-колонка, слияние на бэкенде — без отдельной таблицы «доп. заявок»

**Модуль 3. Две контрольные услуги ТЗ — обе в конструкторе**
- «Приобретение авиатранспорта и вагонов в лизинг» (БРК-Лизинг) — 6 шагов, ~40 полей, DSCR / IRR / казсодержание / KYC (миграция [003](backend/migrations/003_realism_pack.up.sql))
- «Агробизнес: развитие животноводства» — 6 шагов, ~30 полей, 3 расчётных (субсидия / платёж), условная видимость по виду скота, этап 2 = ветзаключение и отчётность (миграция [011](backend/migrations/011_agroanimal_control.up.sql))
- Обе засеяны как `FormSchema` со `stage`, а не захардкожены

**Модуль 4. AI-ассистент**
- `POST /api/ai/generate-form` и потоковый `/generate-form-stream` (SSE) — по описанию Claude возвращает **за один вызов** метаданные (название, категория, орган, ставка/сумма/срок из закрытых списков), полную схему и **разметку этапов** (`stage 1` vs `stage 2`)
- `POST /api/ai/recommend` — подбор 2–3 альтернативных услуг под профиль заявителя (при отказе, при блокирующих рисках, в онбординге)
- Бэкенд [handlers/ai.go](backend/internal/handlers/ai.go) — 437 строк, модель `claude-sonnet-4-6`, прямой HTTPS без SDK

**Модуль 5. Клиентский путь заявителя**
- Вход по ИИН без пароля (имитация eGov IDP), JWT-сессия
- Каталог + публичная страница услуги с **авто-калькулятором**, порождённым из расчётных полей схемы конструктора (админ добавил формулы → на витрине сам появился калькулятор)
- Подача: пошаговая форма из `form_schema`, real-time пересчёт, условные шаги, префилл eGov (бейдж «из eGov»)
- **Предварительная оценка заявителя** на шаге «Проверка»: балл 0–100 по mock eGov + КГД (налоговая дисциплина, выручка и рост, зрелость бизнеса, долговая нагрузка запроса), бэнды **A–D**, предодобряемый лимит, честная подпись «справка, не решение о выдаче» ([prescore.ts](frontend/src/lib/prescore.ts) 299 стр. + [PrescoreCard.tsx](frontend/src/components/PrescoreCard.tsx) 335 стр.)
- Досдача этапа 2, загрузка документов, история, статусы, уведомления

**Модуль 6. Карта проектов, каталог аналитики, обучающие материалы (критерии по 3 балла)**
- **Карта проектов** `/projects-map` (Leaflet) — профинансированные проекты по регионам РК, фильтры по организации/отрасли/статусу (данные статические на конкурсном этапе)
- **Каталог аналитической отчётности** `/analytics` — единый список отчётов/дашбордов дочек с предпросмотром; агрегирует их BI, не дублирует
- **База знаний** — категории, 6 скачиваемых шаблонов документов, 2 интерактивных калькулятора (посчитаны на фронтенде)

**Модуль 7. Продуктовая аналитика программы (сверх ТЗ)**
- Режим «Аналитика» в конструкторе услуги: воронка от просмотра карточки до одобрения с процентом просадки на каждом шаге; наибольшая просадка — красным
- Клик по шагу → drilldown по полю, где отвалились: статистики (p25 / медиана / p75) + детерминированный insight по эвристикам
- Режим «Аудитория» (калькулятор охвата): фильтры по отрасли/региону/размеру/выручке, число подходящих за ~300 мс, персонализированная рассылка `{{full_name}}`/`{{org_name}}`; замыкание с воронкой кнопкой «Перенастроить аудиторию»
- Источники — синтетические профили и события ([audience.go](backend/internal/handlers/audience.go) 287 стр., [funnel.go](backend/internal/handlers/funnel.go) 431 стр.). В проде — реальные события через хуки; **не дублирует** BI дочек

## Слайд 4 — Контрольные кейсы — глубина проработки

Сам факт работающей формы — это минимум. Жюри Байтерек ожидает реалистичности. ТЗ раунда даёт **две** контрольные услуги с bgov.kz — обе собраны в конструкторе как двухэтапные.

**Кейс 1. «Приобретение авиатранспорта и вагонов в лизинг»** (АО «БРК-Лизинг») развёрнут в полную BRK-grade конфигурацию — 6 шагов, ~40 полей, миграция [003_realism_pack.up.sql](backend/migrations/003_realism_pack.up.sql):

1. **Информация о компании** — БИН, ОКЭД, дата регистрации, выручка за 3 года, численность
2. **Предмет лизинга и казсодержание** — авиатранспорт / вагоны / спецтехника, страна происхождения, доля каз. содержания, поставщик
3. **Финансовая модель** — аванс ≥ 15%, сумма лизинга, ставка, **DSCR** (целевой ≥ 1.3), **IRR** (≥ 12%), доля собственного участия (≥ 20%) — всё с автоматическим расчётом по формулам
4. **Обеспечение и оценка** — аккредитованный оценщик из реестра БРК, страховая компания
5. **KYC / ПОД-ФТ** — бенефициар > 25%, источник средств, согласия на ПКБ / ГКБ, ПДн (Закон РК № 94-V), санкционная декларация, согласие на запрос в КГД и ЕНПФ
6. **Документы** — фин. отчётность за 3 года, ТЭО, протокол участников, СТ-KZ при наличии

Это не просто «6 полей и кнопка Submit». Это структура заявки, которая знакома сотрудникам БРК-Лизинг по основной деятельности.

**Кейс 2. «Агробизнес: развитие животноводства»** (миграция [011_agroanimal_control.up.sql](backend/migrations/011_agroanimal_control.up.sql)) — 6 шагов, ~30 полей, 3 расчётных:

1. **Хозяйство и заявитель** — форма (КХ / ТОО / СПК), БИН, ОКЭД, регион
2. **Направление животноводства** — вид скота (КРС / МРС / птица / лошади); условная видимость полей по виду
3. **Параметры проекта** — поголовье, корма, оборудование, строительство/модернизация фермы
4. **Финансовая модель** — сумма, льготная ставка (от 5%), субсидия части затрат (до 50%) — расчётные поля субсидии и платежа
5. **Обеспечение и соответствие** — залог, ветеринарное соответствие
6. **Документы (этап 2)** — ветеринарное заключение, отчётность за 3 года — донос по запросу администратора

Обе услуги двухэтапные: шаги документов помечены `stage 2` и запрашиваются после первичного рассмотрения. Это прямая реализация «многоэтапного сценария» из ТЗ — средствами конструктора, без кода под конкретную услугу.

## Слайд 5 — Тиражируемость: одна архитектура → много мер поддержки

Помимо контрольного кейса, в базу засеяны меры от других институтов развития (миграция [002_seed_services.up.sql](backend/migrations/002_seed_services.up.sql) + 003):

| Орган | Меры |
|---|---|
| АО «БРК-Лизинг» | Лизинг авиатранспорта / вагонов (контрольный кейс) |
| Фонд «Damu» | Гарантирование кредитов МСБ (до 85% / 7 млрд ₸) · Субсидирование ставки ДКБ-2025 (до 6 п.п.) · Грант молодёжного предпринимательства (до 5 млн ₸) |
| «Демеу» | Льготное финансирование МСБ по ставке 6% |
| АО «КазАгроФинанс» / «АгроКапитал» | Льготное кредитование сельхозпроизводителей · Лизинг сельхозтехники |
| Экспортные программы | Страхование экспортных контрактов · Предэкспортное финансирование |
| Прочее | Грант для инновационных стартапов (Seed) и др. |

**Сейчас в БД:** ~16 предзаполненных услуг с реалистичной финансовой моделью каждая.
**Целевая ёмкость архитектуры:** 70+ мер господдержки, перечисленных в техническом задании.

Что важно для жюри: добавление новой меры — это SQL-INSERT в таблицу `services` либо нажатие кнопки в конструкторе. Не Pull Request, не релиз, не работа разработчика. Это и означает «нет потолка по числу услуг».

## Слайд 7 — Команда и статус

**Состав:** один разработчик, full-stack + интеграция AI. Backend (Go), frontend (React/TS), миграции, инфраструктура (Docker), интеграция Anthropic Claude — реализованы одним человеком.

**Стадия:** Working MVP. Система запускается командой `make up`, проходит двусторонний сценарий жюри (AI-генерация формы → разметка этапов → публикация → подача заявителем с префиллом и предоценкой → **дозапрос документов** админом → досдача этапа 2 → одобрение) за ~4–5 минут.

**Тестовые входы для жюри:**
- Администратор: ИИН `000000000000`, любое имя
- Пользователь: любые 12 цифр в ИИН, любое имя
- URL: `http://localhost:3000` после `make up`

**Что подтверждено в коде и БД:**
- ~16 услуг засеяно с реалистичными формами (после дедупликации лизинга миграцией 007)
- Обе контрольные услуги ТЗ — лизинг (6 шагов, DSCR/IRR/казсодержание/KYC) и агроживотноводство (6 шагов, 3 расчётных, условная видимость) — двухэтапные
- Двухэтапный цикл `docs_requested → stage2` — рабочий, эндпоинт `SubmitStage2` с мёржем `form_data`
- AI-генерация формы + метаданных + этапов из описания — рабочая, тестировалась (есть e2e-тесты)
- Предоценка заявителя A–D и eligibility-движок — по mock eGov+КГД

## Слайд 9 — Следующий шаг

У жюри три уровня участия — от лёгкого к серьёзному. Прошу выбрать один.

**1. Быстрая верификация (30 секунд, прямо сейчас).**
Назовите любую меру господдержки из списка 70+. Я создам её форму на сцене через AI-генерацию + ручную правку и опубликую за время разговора. Это снимает вопрос «работает ли no-code-движок на произвольной мере».

**2. Интро к куратору в Байтерек / Damu / БРК-Лизинг.**
Контакт человека, который курирует приём заявок по одной из мер. Цель — двухнедельный технический созвон, согласовать пилотный объём (одна мера, один регион) и формат интеграции.

**3. Победа в номинации хакатона + поддержка пилотной мобилизации.**
Призовое место даёт легитимность для входа в институт развития. Дальше — 60–90 дней до пилотного запуска одной меры на ограниченном контуре (одна категория получателей, один регион). Финансирование и доработку беру на себя.

Перспектива после пилота: пилот → регион → республиканское покрытие. Конкретные сроки этой части обсуждаются после результатов пилота, без обязательств жюри сейчас.
