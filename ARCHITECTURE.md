# Архитектурный подход — Qoldau AI (ЕППБ)

**Единый портал поддержки бизнеса.** Хакатон Astana Hub × АО «НИХ «Байтерек».
Подача решения — 12.07.2026, финалисты — 14.07.2026, Demo Day — 16.07.2026.

## 1. Ключевая идея

Требование ТЗ — не строить очередной сайт с формой под одну услугу, а дать холдингу
инструмент, которым любую из 70+ мер поддержки можно **описать как данные**, а не
запрограммировать руками.

В основе — единая модель `FormSchema`, которая хранится в PostgreSQL (JSONB,
колонка `services.form_schema`) и описывает шаги, поля, формулы, условия видимости и
этапность подачи заявки. Один фронтенд-компонент (`FormRenderer`) умеет отрисовать
**любую** услугу, подчиняясь этой схеме — от простой анкеты на грант до многошаговой
заявки на лизинг с расчётными полями и документами по запросу.

Итог: чтобы вывести новую меру поддержки на портал, не нужен релиз кода — администратор
или бизнес-аналитик собирает её в конструкторе (вручную или через AI-генерацию) и
публикует. Это прямо снимает риск «услуга захардкожена», обозначенный как критичный в
условиях конкурса.

## 2. Технологический стек

**Frontend** — React 18.2 + TypeScript + Vite 5.2; `@tanstack/react-query` 5.29 (серверное
состояние, кэш, инвалидация после мутаций); `zustand` 4.5 (auth-стор); `axios` 1.6
(HTTP-клиент с интерцептором авторизации); `react-router-dom` 6.23; `react-hook-form` +
`zod` (валидация форм за пределами конструктора); `expr-eval` (безопасное вычисление
формул `calculated`-полей без `eval`); `leaflet` + `react-leaflet` (интерактивная карта
проектов); `react-joyride` (онбординг-тур по конструктору форм); `@dnd-kit/*` в
зависимостях (заложено под drag-and-drop реордер полей конструктора).

**Backend** — Go 1.22, роутер `chi` v5.1 (+ `chi/cors`, middleware `Logger`/`Recoverer`/
`RealIP`), `sqlx` 1.3 поверх `lib/pq`, JWT (`golang-jwt/jwt` v5), миграции —
`golang-migrate` (применяются автоматически при старте сервера), Redis 7 (клиент
подключён, зарезервирован под кэш/сессии — на данный момент сбой подключения только
логируется предупреждением, не блокирует старт).

**Хранилище** — PostgreSQL 16 (Alpine). 11 миграций (`001_init` … `011_agroanimal_control`):
базовая схема и сиды → каталог услуг → правила eligibility → аудиторный профиль
заявителя (`005_audience`) → воронка/аналитика событий (`006_funnel`) → условия
программы (`008_program_terms`) → статус `docs_requested` и двухэтапность
(`009`, `010_two_stage`) → вторая контрольная услуга «Агробизнес: развитие
животноводства» (`011`). Оба контрольных кейса конкурса — «Приобретение вагонов
в лизинг» и «Агробизнес животноводство» — засеяны как многошаговые двухэтапные
услуги (шаги документов помечены `stage 2`), то есть собраны средствами
конструктора, а не реализованы кодом.

**AI** — Anthropic Claude API, модель `claude-sonnet-4-6`, вызывается напрямую по HTTPS
из Go-хендлера (без промежуточного SDK) в трёх сценариях: генерация `form_schema` и
метаданных услуги, потоковая (SSE) генерация для UI конструктора, подбор
альтернативных услуг заявителю.

**Деплой** — Docker Compose: `postgres:16-alpine`, `redis:7-alpine`, backend
(многоступенчатая сборка Go-бинаря), frontend (сборка Vite → статика, раздаётся через
nginx, который также проксирует `/api`). Каждый сервис — отдельный контейнер с
healthcheck-зависимостями (`backend`/`frontend` ждут готовности postgres/redis).

## 3. Модель FormSchema и no-code конструктор

```
FormSchema { steps: FormStep[] }
FormStep   { id, title, fields: FormField[], condition?, stage? }
FormField  { id, type, label, placeholder?, required?, options?,
             mask?, formula?, readonly?, accept?, prefill_from?, condition? }
```

Типы полей: `text | textarea | number | currency | select | multiselect | date | file |
calculated | checkbox | radio`.

Ключевые механики, зашитые в схему как данные, а не в код конкретной услуги:
- **`condition`** (на шаге или поле) — `{ field_id, operator: equals|not_equals|
  greater_than|less_than, value }`; движок (`FormRenderer`) фильтрует видимые шаги/поля
  на каждый ре-рендер через `checkCondition`, то есть ветвление сценария подачи заявки
  описывается администратором, а не программистом.
- **`formula`** — JS-подобное выражение над `id` других полей, разбирается и
  вычисляется библиотекой `expr-eval` (без `eval()`, безопасно). Движок строит план
  зависимостей (`buildCalcPlan`) один раз при загрузке схемы и на каждое изменение
  пересчитывает только «грязные» calculated-поля — данные обновляются в реальном
  времени по мере ввода.
- **`prefill_from`** (`'egov.xxx'`) — автозаполнение значением из mock eGov API;
  поле остаётся редактируемым, но помечается бейджем «из eGov».
- **`stage`** на шаге — этапность подачи (см. раздел 4).

Конструктор — `frontend/src/pages/admin/ServiceFormPage.tsx` (~1800 строк) — даёт
администратору/автору собрать услугу визуально: холст шагов и полей, панель
метаданных (название, категория, орг-название, ставка/сумма/срок), переключатель
stage 1/2 на каждом шаге, кнопки «Сохранить черновик» / «Опубликовать»
(`POST /api/services`, `POST /api/services/:id/publish`). Дополнен онбординг-туром
(`react-joyride`) для демонстрации жюри за ~60 секунд.

Важно: конструктор и рендерер работают с одной и той же структурой `FormSchema` —
что администратор собрал (руками или через AI), то заявитель без доработки кода
увидит в личном кабинете (`FormRenderer` в `pages/cabinet/ApplyPage.tsx`).

## 4. Двухэтапный сценарий заявки

Статусная модель (`models.ApplicationStatus`): `draft → submitted → in_review →
[docs_requested] → approved | rejected`.

Таблица `applications` несёт поля `stage int` и `request_message string` (миграция
`010_two_stage`), помимо стандартных `form_data JSONB`, `status`.

Флоу, реализованный в `handlers/applications.go`:

1. **Первичная подача.** Заявитель проходит шаги с `stage = 1` (или без поля stage —
   трактуется как 1) через `FormRenderer`, `POST /api/applications` создаёт заявку со
   статусом `submitted`, `form_data` — JSONB со значениями только видимых на тот момент
   полей.
2. **Запрос доп. данных.** Администратор через `PUT /api/applications/:id/status` с
   `status = docs_requested` и `message` — сообщением, что именно нужно донести.
   Хендлер атомарно пишет `request_message` вместе со статусом и создаёт уведомление
   заявителю с этим текстом.
3. **Дозаполнение.** Заявитель видит статус и `request_message` в личном кабинете,
   заполняет шаги со `stage = 2` (обычно — загрузка документов/расширенные данные) и
   отправляет `POST /api/applications/:id/stage2`. Хендлер `SubmitStage2` проверяет,
   что заявка действительно в `docs_requested` и принадлежит заявителю, **мёржит**
   новые ключи поверх существующего `form_data` (ключи stage 2 перезаписывают
   совпадающие ключи stage 1), переводит `stage → 2`, `status → in_review`.
4. Далее — обычный жизненный цикл: `approved`/`rejected` с уведомлением заявителю.

Это прямая реализация контрольного кейса конкурса («сначала общая заявка, затем
донос документов по запросу») — без отдельной таблицы «доп. заявок»: одна запись
`applications`, одна JSONB-колонка, слияние на уровне бэкенда.

## 5. AI-компонент

Реализован в `backend/internal/handlers/ai.go`, три эндпоинта:

- **`POST /api/ai/generate-form`** и потоковый **`/api/ai/generate-form-stream`** (SSE,
  `X-Accel-Buffering: no` для работы за nginx) — по текстовому описанию услуги
  (админ пишет свободным текстом) Claude возвращает **одним вызовом** и метаданные
  программы (`title`, `description`, `category`, `org_name`, `interest_rate`,
  `max_amount`, `max_term_months` — строго из закрытых списков категорий/организаций,
  заданных в системном промпте), и полный `form_schema` с шагами/полями/формулами/
  условиями. Системный промпт отдельно инструктирует модель различать `stage 1`
  (первичная подача) и `stage 2` (донос документов после предварительного одобрения) —
  то есть AI осведомлён о двухэтапном сценарии и сам расставляет `stage` на шагах.
  Потоковый вариант используется в конструкторе для UX (прогресс-бар генерации,
  анимация появления шагов); синхронный — как fallback/API.
- **`POST /api/ai/recommend`** — подбор 2–3 альтернативных услуг под профиль заявителя
  (данные mock eGov + mock КГД, опционально причина отказа), с обоснованием на основе
  выручки, возраста бизнеса, налоговой задолженности, реестра риска, ОКЭД. Используется
  при отказе по заявке, при блокирующих рисках до подачи, в публичном онбординге.
  Список услуг для матчинга запрашивается без `form_schema` (экономия токенов).

Практическая польза AI в MVP — не демонстрационная, а рабочая часть конструктора
(снимает с администратора рутинную сборку схемы вручную) и часть клиентского пути
(снижает число отказов «в пустоту», подсказывая альтернативу).

## 6. Масштабируемость и разделение ответственности

- Новая услуга = новая строка `services` с `form_schema`/`eligibility_rules` в JSONB —
  без миграций и без деплоя. Масштабирование на 70+ мер поддержки — вопрос
  наполнения каталога, а не разработки.
- Роли: `user` (заявитель) / `author` (создаёт и редактирует услуги, не публикует и не
  меняет статусы заявок) / `admin` (публикация, статусы заявок, аналитика, рассылки).
  Разграничение — на уровне middleware (`RequireRole`), не на уровне бизнес-логики
  конкретной услуги.
- Портал **не заменяет** BPM-системы дочерних организаций и единую CRM холдинга.
  Явная граница: ЕППБ — это (а) единая точка входа и клиентский путь заявителя
  (каталог → форма → статус → уведомления), (б) конструктор, снимающий барьер
  «под каждую услугу — своя команда разработки». Решение о выдаче финансирования,
  скоринг, бэк-офис — остаются в системах дочерних организаций; портал передаёт им
  заявку и получает статус.
- Специализированная аналитика и отчётность дочерних организаций **не дублируются** —
  подключаются через каталог ссылок/embedding (раздел 8), чтобы не плодить
  параллельную BI-систему поверх уже существующих.

## 7. Готовность к интеграциям

На этапе конкурса внешние интеграции реализованы как контролируемые заглушки с чётко
обозначенной границей, куда встанет реальный сервис:

| Заглушка сейчас | `backend/internal/handlers/mock.go` | Куда встанет в проде |
|---|---|---|
| `GET /api/mock/egov/:iin` | детерминированные ФИО/БИН/адрес по ИИН (спецданные для сид-админа `000000000000`, дефолт для остальных) | eGov IDP / реестр ФЛ-ЮЛ — авторизация и автозаполнение профиля |
| `GET /api/mock/kgd/:bin` | псевдо-случайные (по последней цифре БИН) данные налогового досье: режим налогообложения, выручка за 3 года, ФОТ, налоговая задолженность, ОКЭД, `compliance_status` — по форме, повторяющей `cabinet.salyk.kz` | Комитет государственных доходов — проверка налоговой добросовестности при скоринге eligibility |
| `POST /api/mock/eish/submit` | возвращает фиктивный `external_id` вида `EISH-2026-XXXXX` и статус `accepted` | Единая информационная шина холдинга — передача заявки в BPM дочерней организации |

Авторизация в MVP — JWT по паре `{iin, full_name, org_name?}` без пароля (mock
eGov-вход), что соответствует условиям конкурса («допускается упрощённая имитация
входа») и оставляет ясный порт для подключения настоящего eGov IDP (OAuth/OIDC) без
изменения структуры токена на фронтенде. ЭЦП (подписание итоговой заявки/договора)
и полноценная передача в BPM/ЕИШ — не реализованы, но архитектурно отделены:
обе операции естественно ложатся как шаг после `approved` в жизненном цикле заявки,
не требуя изменений в `FormSchema`.

## 8. Дополнительные модули

- **Карта проектов** (`frontend/src/pages/ProjectsMapPage.tsx`, Leaflet/`react-leaflet`) —
  интерактивная карта профинансированных/реализуемых проектов по регионам РК с
  фильтрами по организации, отрасли, статусу; координаты — реальные центры областей с
  джиттером на проект. **На конкурсном этапе данные статические** (массив в коде), без
  бэкенд-эндпоинта — демонстрирует UX-концепцию прозрачности, для прода требует
  вынесения в таблицу/API.
- **Каталог аналитической отчётности** (`AnalyticsCatalogPage.tsx`) — единый список
  готовых отчётов/дашбордов дочерних организаций (тип, период, источник, формат
  web/pdf/embed) с фильтрами и предпросмотром. Осознанно не дублирует BI дочерних
  организаций — только агрегирует ссылки/embedding на их существующую отчётность.
  Также пока со статическими демо-данными.
- **Раздел материалов и инструментов** (`KnowledgePage.tsx`) — база знаний
  (категории: с чего начать, подача заявки, документы, ЭЦП/eGov, финансирование, FAQ),
  шаблоны документов и калькуляторы (кредитный калькулятор — аннуитетный платёж по
  сумме/ставке/сроку, посчитан на фронтенде без обращения к бэкенду).

## Соответствие требованиям ТЗ

- **Услуги через no-code конструктор, а не хардкод** — `FormSchema` в JSONB +
  единый `FormRenderer`/`ServiceFormPage`; новая услуга не требует релиза кода
  (разделы 1, 3).
- **Условная логика и расчётные поля** — `condition` (шаг/поле) и `calculated` +
  `formula`, пересчёт в реальном времени через `expr-eval` (раздел 3).
- **Двухэтапная подача с запросом доп. документов** — статус `docs_requested`,
  `request_message`, `stage`, эндпоинт `stage2` с мёржем `form_data` (раздел 4).
- **Оба контрольных кейса реализованы через конструктор** — «Приобретение
  авиатранспорта и вагонов в лизинг» (6 шагов) и «Агробизнес: развитие
  животноводства» (6 шагов, расчётные поля субсидии/платежа, условные поля по виду
  скота) засеяны как `FormSchema` с этапом 2, а не захардкожены (разделы 3, 4).
- **AI как практический инструмент, не витрина** — генерация `form_schema` и метаданных
  услуги по описанию (в т.ч. потоково), рекомендация альтернативных услуг заявителю
  (раздел 5).
- **Упрощённая авторизация через eGov** — mock `GET /api/mock/egov/:iin` + JWT без
  пароля, с явным портом под реальный eGov IDP (раздел 7).
- **Готовность к интеграции с системами холдинга** — мок-заглушки eGov/КГД/ЕИШ с чётко
  описанной границей замены на боевые интеграции через единую интеграционную шину
  (раздел 7).
- **Не конкурирует с существующими системами (bgov.kz, BPM дочерних организаций)** —
  явное разделение ответственности: клиентский путь и конструктор — в ЕППБ, скоринг и
  бэк-офис — в системах дочерних организаций; аналитика дочерних организаций
  агрегируется, а не дублируется (разделы 6, 8).
- **Масштабируемость на 70+ мер поддержки, множественные роли/организации** — модель
  данных не завязана на конкретную услугу; роли `user/author/admin` разграничивают
  создание, публикацию и рассмотрение (раздел 6).
